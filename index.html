Как подружить 15 AI-агентов и не сойти с ума

«У каждой проблемы есть решение, которое простое, изящное и неправильное.» — Х. Л. Менкен

Вы добавили пятого агента в команду — и всё сломалось. Десятого — сломалось снова. К пятнадцатому приходит понимание: проблема не в моделях, не в промптах, не в людях. Проблема — в процессе.

Мы прошли этот путь. 15 AI-агентов — разработчики, тестировщики, девопсы, аналитики, — которые должны были работать как команда, а работали как толпа. Задачи терялись, артефакты исчезали, «готово» означало «я нажал кнопку», а не «оно работает».

Эта статья — о том, как мы превратили хаос в процесс, какие протоколы и паттерны реально работают, а какие — красивые слайды без реализации.

1. Как всё сломалось

Начиналось классически. Один агент — ChatGPT с плагинами — делал всё сам: планировал, писал код, тестировал, деплоил. Работало, пока задач было пять.

Потом появился второй агент — фронтендер. Потом третий — девопс. К десятому стало ясно: каждый агент живёт в своём мире. Контекст теряется на каждом переходе. Бэкендер написал API — фронтендер не знает какое. Тестировщик проверил старую версию. Девопс задеплоил то, что никто не ревьюил.

Но главная проблема была не в коммуникации. Она была в процессе.

Фантомный Done

Мы обнаружили, что задачи помечались как выполненные, но артефактов не было. Задача «Реализовать авторизацию» — статус Done. Работающей авторизации — нет. Как так? Системе неоткуда проверить, что результат существует и соответствует критериям.

Gatekeeper на bash

Мы написали «сторожа» — bash-скрипт, который проверял, можно ли перевести задачу на следующий этап. Цепочка if/then с вызовами GitHub API и проверкой полей. Bash прекрасен для склеивания команд, но не для бизнес-логики: нет типизации, сложно обрабатывать нестандартные ситуации, каждое новое правило — ещё один вложенный if. Процесс меняется — скрипт ломается, и никто не замечает.

Верификаторы без зубов

Проверки качества искали текстовые признаки вместо архитектурных контрактов. Верификатор искал строку TODO в коде — и находил. Но если вся архитектура нарушена (модуль А зависит от Б, Б зависит от А) — верификатор ничего не замечал. Строки-то правильные.

Вывод: пятнадцать агентов без формализованного процесса — как пятнадцать программистов без методологии. Каждый делает что-то полезное, но вместе — хаос. И это не только про разработку: продажи, саппорт, аналитика — любой процесс с фазами и артефактами подвержен той же проблеме.

2. Что уже есть: протоколы

Прежде чем писать свой велосипед, стоит посмотреть что уже существует. Индустрия стандартизировала несколько протоколов для взаимодействия агентов — и они решают ровно те проблемы, с которыми мы столкнулись.

MCP (Model Context Protocol) — Anthropic

Зачем: Агент ↔ Инструменты и данные

MCP — самый зрелый протокол в пространстве. Официальные SDK для Python и TypeScript, широкая поддержка. Определяет три сущности:

  • Tools — действия, которые агент может вызвать (создать issue, обновить борд)
  • Resources — данные, которые агент может читать (файлы, API-ответы)
  • Prompts — шаблоны поведения

Для нас MCP заменяет самодельные CLI-скрипты для работы с GitHub. Вместо bash-цепочек с gh api graphql — типизированные инструменты с контрактом. Официальный GitHub MCP Server уже умеет CRUD issues, Projects V2, PR и Actions.

Минус: ~150K токенов на загрузку всех tool definitions. Решение — загружать только нужные.

A2A (Agent-to-Agent) — Google → Linux Foundation

Зачем: Агент ↔ Агент

A2A решает проблему передачи работы между агентами. Ключевые концепции:

  • Agent Card — JSON-описание возможностей агента (навыки, протоколы, endpoint)
  • Task — единица работы, передаваемая между агентами
  • Message — коммуникация внутри Task (артефакты, статус, вопросы)

Для нас A2A заменяет bash-gatekeeper: обязательные поля в Task payload = формальный контракт на переход между фазами. Оркестратор передаёт задачу дизайнеру — и в payload обязательно есть требования. Нет требований — задача не принимается. Это gatekeeper без строчки кода.

Коммуникация: JSON-RPC over HTTP/SSE — стандартные веб-технологии.

ACP (Agent Communication Protocol) — IBM → Linux Foundation

ACP был REST-based альтернативой A2A. Сейчас официально влился в A2A — отдельного протокола больше нет, его концепции интегрированы.

AGP (Agent Gateway Protocol) — Cisco/agntcy

Иерархическая маршрутизация между агентами — надстройка над A2A для масштабирования на тысячи агентов и межорганизационное взаимодействие. Спецификация draft, референс-имплементация минимальная. 15 агентов — A2A достаточно. На будущее, если понадобится мониторинг агентов разных организаций.

3. Как организовать команду: паттерны взаимодействия

Протоколы — это транспорт. Но кто кому говорит, когда и что — это паттерн. От выбора паттерна зависит всё.

Monolithic — один агент делает всё

Один агент получает задачу и выполняет все шаги сам. Просто, но контекстное окно переполняется, токены горят, ошибка на одном этапе ломает всё. Подходит для прототипов и разовых задач. Не масштабируется.

Orchestrator + Workers — менеджер + исполнители

Центральный агент разбивает задачу, делегирует, собирает результаты. Чёткая ответственность, но оркестратор — единая точка отказа. Если менеджер «засыпает» — всё стоит.

Pipeline — последовательная передача

Агенты выстроены в цепочку: выход одного = вход следующего. Предсказуемо, соответствует SDLC, но нет параллелизма. Один застрял — весь конвейер стоит.

Event-driven — реакция на события

Агенты подписаны на события и реагируют асинхронно. Слабая связанность, но сложно отладить, нет гарантии порядка.

Swarm/Mesh — децентрализованная координация

Агенты общаются напрямую, без оркестратора. Нет единой точки отказа, но непредсказуемо, токены тратятся на «обсуждение», может зациклиться.

State Machine / Workflow — формальный автомат

Процесс описан как конечный автомат. Переходы — детерминистические guards (код, не LLM). 100% воспроизводимость, ноль токенов на оркестрацию, невозможно нарушить процесс. Но жёсткость — нельзя «подправить на ходу».

Что выбрать

Для SDLC и продакшн-процессов — State Machine для переходов (детерминистические guards) + Orchestrator+Workers для исполнения (менеджер делегирует агентам). Pipeline не даёт параллелизма, Swarm — хаос, Monolithic — не масштабируется.

5. Наш подход: SDLC как конечный автомат

Мы выбрали комбинацию State Machine + Orchestrator. Вот как это работает:

Переходы — детерминистические

Backlog → Requirements → Design → Implementation → Review → QA → Deploy → Done
   │         │            │          │             │       │      │
   └──[gate]─┘──[gate]───┘──[gate]──┘──[gate]────┘──[gate]─┘──[gate]─→

Каждый переход — guard-функция, которая проверяет что артефакт существует:

  • Requirements → Design: есть документ с требованиями
  • Design → Implementation: есть дизайн-документ
  • Implementation → Review: есть открытый PR
  • Review → QA: PR смержен
  • QA → Deploy: вердикт ✅ от тестировщика
  • Deploy → Done: задеплоено + QA подтверждено

Guard — это код (exit 0 / exit 1), не LLM. Ноль токенов, 100% воспроизводимость, невозможно обойти.

Исполнение — оркестратор + исполнители

Оркестратор (менеджер) получает задачу, определяет фазу, делегирует агенту с нужной ролью:

  • Requirements → Борис (аналитик)
  • Design → Вика (дизайнер)
  • Implementation → Дима или Настя (разработчики)
  • Review → Олег (Scrum Master, назначает ревьюера)
  • QA → Артём (тестировщик)
  • Deploy → Катя (девопс)

Каждый исполнитель делает работу, верифицирует результат, возвращает оркестратору. Оркестратор проверяет guard и двигает задачу дальше.

Фантомный Done — больше не возможен

Задача не попадёт в Done без:

  1. Смерженного PR (проверяется GitHub API)
  2. QA-вердикта ✅ (проверяется последним комментарием на issue)
  3. Факта деплоя (проверяется的健康-check endpoint)

Всё — код, не промпты. LLM может ошибиться — guard нет.

6. Что мы узнали

Процесс > промпты

Хороший промпт одному агенту — это 100 строк. Хороший процесс для пятнадцати — это 5 guard-функций и таблица ролей. Процесс воспроизводим, промпт — нет.

Не пиши свой HTTP

Bash-gatekeeper, самодельные верификаторы, собственный формат передачи контекста — это велосипед. MCP стандартизирует доступ к инструментам. A2A стандартизирует передачу задач. Используй их.

Guard — код, не LLM

Каждый раз когда LLM принимает решение о переходе между фазами — это риск. LLM может «забыть» проверить артефакт, может «поверить» что задача готова. Код — не забывает.

Делегирование = конкретная задача + верификатор

Абстрактная задача («напиши тесты») → агент блуждает, жжёт токены, таймаутится. Конкретная задача («проверь endpoint /api/health — curl вернёт 200, лог содержит JSON с level=info») → агент делает и верифицирует.

Один точечный запрос лучше одного широкого

«Проверь все 800 задач» = rate limit. «Проверь задачу #139» = один вызов API. Скрипт уже собрал данные — читай кэш, не ходи в API.

7. Что дальше

Мы описали подход для 15 агентов. Когда их станет 50 или 500 — понадобятся новые решения: AGP для межорганизационной маршрутизации, персистентные сессии вместо ephemeral, формальные контракты вместо ad-hoc делегирования.

Но фундамент тот же: процесс из фаз, guards на переходах, артефакты как пропуск, код как верификатор. Это работает для 5 агентов, для 15, и будет работать для 500.
