Как подружить 15 AI-агентов и не сойти с ума

Архитектура агентов Veai на базе MCP + A2A

Версия 2.0 — 08.06.2026
Статус: Ещё не поздно развернуть

1. Проблема — или «почему 15 агентов хуже одного»

Команда из 15 AI-агентов, которые закрывают задачи сами — звучит как утопия. На практике выглядит как 15 котов в мешке, каждый из которых уверен, что он единственный, кто прав. Один создаёт issue, второй его закрывает «по приколу», третий пишет код, который четвёртый тут же переписывает, а пятый только что запушил в main без ревью. Welcome to агентный хаос.

Вот с чем мы живём сейчас, когда SDLC формализован через самописные bash-скрипты:

  • Gatekeeper на bash — хрупкий как китайский фарфор, сложно расширять, типизация? Не слышали
  • Верификаторы на grep/python — проверяют строки, а не контракты. Ищут «TODO», а не «нарушена ли архитектура»
  • Контекст между агентами — испаряется быстрее утреннего кофе. Один спроектировал, другой не заметил, третий переписал
  • Нет стандартизации — каждый агент придумывает свою логику перехода, как будто мы пишем протоколы с нуля ради удовольствия
  • Фантомный Done — задача помечена «Готово», но никто не проверил артефакты. Потому что нечем.

Индустрия уже всё придумала. Писать HTTP на bash в 2026 году — это как копать огород ложкой, когда экскаватор стоит в гараже.

2. Язык, на котором говорят порядочные агенты — MCP, A2A и другие

2.1 MCP — агент тянется к инструменту (как ребёнок к розетке)

Назначение: Агент ↔ Инструменты/Данные. Протокол, который говорит агенту: «Вот чем ты можешь пользоваться, руками не трогай».

Зрелость: ⭐⭐⭐⭐⭐ — самый взрослый протокол на рынке. Официальные SDK на Python и TypeScript, широкая индустриальная поддержка, сообщество растёт как на дрожжах.

Ключевые концепции: - Tools — действия, которые агент может вызвать (создать issue, обновить борд, поставить лайк, поменять свечку — фантазия ограничена только SDK) - Resources — данные для чтения (файлы, API-ответы, чужие ошибки) - Prompts — шаблоны поведения, чтобы не объяснять агенту каждый раз «как надо»

Что закрывает у нас: - GitHub MCP Server (официальный) — CRUD issues, Projects V2, PR, Actions - Отправляет на пенсию: CLI-скрипты управления бордом - Отправляет на пенсию: прямые gh api graphql вызовы - Отправляет на пенсию: кэш (локальный кэш — MCP server справится сам)

Цена вопроса: ~150K токенов на загрузку всех tool definitions (по данным Anthropic). Решение: грузить не все подряд, а только нужные toolset-ы. Как шведский стол — бери, что съешь.

2.2 A2A — когда агентам есть что обсудить

Назначение: Агент ↔ Агент. О том, как один ИИ говорит другому «я закончил, принимай работу».

Зрелость: ⭐⭐⭐ — спецификация стабильна, референс-имплементации есть, но экосистема ещё щенячьего возраста.

Словарь терминов: - Agent Card — визитка агента (JSON): «меня зовут Борис, я умею писать код, не умею пить кофе» - Task — единица работы, которую агент передаёт другому - Message — что они говорят друг другу внутри задачи (артефакты, статус, пассивная агрессия) - Push Notifications — «я всё» без необходимости дёргать

Что даёт нам: - Передачу артефактов между фазами SDLC (Planning → Design → Implementation → «ой, опять переделывать») - Контракты: обязательные поля в Task payload — gatekeeper без единой строчки на bash - Оркестрацию: оркестратор как client agent, команда как remote agents - Заменяет: gatekeeper в CLI-скрипты (проверка обязательных полей до передачи)

На проводах: JSON-RPC over HTTP/SSE — ничего экзотического, стандартные веб-технологии.

2.3 ACP — умер, но не до конца

Статус: ⚰️ ACP официально влился в A2A. RIP.

ACP был REST-based альтернативой A2A (без JSON-RPC, попроще, для бедных). Сейчас его идеи живут внутри A2A. Реализовывать отдельно — как искать вторую половинку для уже женатого человека.

2.4 AGP — для тех, кого 15 агентов не впечатляют

Назначение: Иерархическая маршрутизация между роями агентов. AGP — это когда A2A уже мало, а хочется управлять тысячами.

Зрелость: ⭐⭐ — черновик спецификации, референс-имплементация на уровне «а давайте попробуем».

Когда нужно: Когда у вас 15 агентов — A2A хватит за глаза. AGP актуален, когда агентов становится 50+, и они разбросаны по разным организациям.

На заметку: Если Veai Control начнёт мониторить агентов разных компаний — AGP станет актуален. Пока — читаем, запоминаем, откладываем.

3. Как агенты «договариваются» — 7 способов, от монолита до хаоса

Протоколы — это грамматика. Архитектура — это то, о чём агенты говорят и кто из них главный. Выбор паттерна — как выбор формы правления: каждый хорош, но не каждый переживёт встречу с 15-ю экзистенциально озабоченными AI.

3.1 Monolithic — «один в поле воин»

┌──────────────────────────────┐
│         Мега-агент          │
│  планирует + пишет + тести  │
│  рулит + деплоит + пишет    │
└──────────────────────────────┘

Как работает: Одна LLM-сессия получает задачу и пашет одна — от требований до деплоя. Контекст един, память едина, конфликтов нет (кроме тех, что в голове у самой LLM). Как тот коллега, который тащит весь проект, а потом выгорает.

Плюсы: Простота. Нет проблем с передачей контекста — он и так один. Нет оверхеда на координацию. Минусы: Контекстное окно забивается быстрее, чем электронная почта после отпуска. Токены сгорают на каждый шаг. Ошибка на раннем этапе убивает всё — и никто не подстрахует.

Когда подходит: Прототипы, разовые задачи, скрипты на вечер. ChatGPT с плагинами — классический монолит. Когда нет: Как только процесс переваливает за 3 фазы или требует разных компетенций — это уже не монолит, а ишак, нагруженный верблюжьей работой.

3.2 Orchestrator + Workers — «генерал и солдаты»

        ┌─────────────┐
        │  Оркестратор │
        └──┬──┬──┬──┬─┘
           │  │  │  │
        ┌──┘  │  │  └──┐
        ▼     ▼  ▼     ▼
      [backend-разработчик][frontend-разработчик][DevOps-инженер][QA-инженер]

Как работает: Центральный оркестратор («менеджер-тиран») получает задачу, дробит на куски, раздаёт исполнителям, собирает результаты, недовольно морщится, просит переделать. Как хороший мидл-менеджер, только без дневных собраний.

Плюсы: Чёткая ответственность. Контекст изолирован — каждый знает свою часть. Исполнителей можно менять, не ломая архитектуру. Минусы: Оркестратор — point of failure. Токены тратятся на координацию, а не на созидание. Если менеджер «засыпает» — вся команда стоит в позе «ждун».

Когда подходит: Команды с ролями, SDLC-процессы, классическое разделение труда. OpenClaw — сюда. Когда нет: Когда задач больше 10 — менеджер захлебнётся, как секретарша в час пик.

3.3 Pipeline — «конвейер и никакой самодеятельности»

[BA] → [Дизайнер] → [Разработчик] → [QA] → [Деплой]
  │        │            │          │        │
  ▼        ▼            ▼          ▼        ▼
 reqs    design       code      verdict  release

Как работает: Агенты выстроены строго по порядку. Output одного = вход следующего. Никаких прыжков через этапы — строго как на заводе Ford, только вместо автомобилей — pull request’ы.

Плюсы: Предсказуемость. Легко аудировать — каждый шаг виден. Естественное соответствие SDLC (BA → Дизайн → Разработка → QA → Деплой). Минусы: Никакого параллелизма. Если QA застрял на «проверяю, подождите» — вся цепочка замерла. Ветвлений нет и не предвидится.

Когда подходит: Линейные процессы, где шаги строго последовательны. LangChain Chains — класcический пайплайн. Когда нет: Как только задача допускает «а давайте одновременно» или «а если условие» — пайплайн ломается.

3.4 Event-driven — «свобода, равенство, братство и pub/sub»

  [GitHub webhook] ──→ [Агент-обработчик]
  [Крон heartbeat] ──→ [Агент-проверщик]
  [PR создан]     ──→ [Агент-ревьюер]

Как работает: Агенты не получают команд — они подписаны на события и реагируют, когда что-то происходит. Как подписчики на Telegram-канале, только без мемасиков.

Плюсы: Слабая связанность — можно добавлять агентов пачкой, никого не трогая. Работает асинхронно. Минусы: Попробуй отладь, если не знаешь, кто на что подписан. События могут теряться. Порядок? Не, не слышали.

Когда подходит: Мониторинг, алерты, реактивные проверки. Cron-задачи (OpenClaw) — типичный event-driven. Когда нет: Там, где «сначала А, потом Б, и точка» — не для событийной архитектуры.

3.5 Swarm — «толпа на площади»

     [Агент A] ⇄ [Агент B]
        ⇕           ⇕
     [Агент C] ⇄ [Агент D]

Как работает: Агенты общаются напрямую, как участники шумного митинга. Нет начальника, нет плана, есть только общее возбуждение и уверенность, что «как-нибудь само решится». Спойлер: не решается.

Плюсы: Нет единой точки отказа. Устойчив к сбоям — если один упал, остальные продолжают спорить. Минусы: Непредсказуемость. Токены тратятся на обсуждение того, «а что бы нам обсудить». Может зациклиться и уйти в бесконечный диалог о смысле жизни. Нет гарантии, что хоть что-то будет сделано.

Когда подходит: Брейнштормы, исследования, творческие задачи. AutoGen group chat — чистый swarm. Когда нет: Продакшн-процессы, SLA, дедлайны. Forge — определённо нет.

3.6 Hierarchical — «корпоративная лестница»

           [Главный менеджер]
          ┌──────┴──────┐
     [Отряд A]      [Отряд B]
     ┌──┴──┐        ┌──┴──┐
   [А1] [А2]     [B1] [B2]

Как работает: Менеджеры делегируют подменеджерам, те — исполнителям. Иерархия как в любой уважающей себя корпорации: сверху вниз, приказы не обсуждаются.

Плюсы: Масштабируется — можно добавить ещё один уровень менеджмента (о ужас). Каждый уровень управляет своей зоной. Чёткая ответственность. Минусы: Задержка на каждом этаже иерархии. Токены тратятся на «менеджер сказал подменеджеру, который сказал супервайзеру». Сложно координировать кросс-отрядные задачи — слишком много согласований.

Когда подходит: Большие команды (10+ агентов) с разными доменами. CrewAI с менеджерами — чистый hierarchical. Когда нет: Маленькие команды и задачи, которые требуют междоменной координации.

3.7 State Machine — «железобетонный формализм»

Backlog → Requirements → Design → Implementation → Review → QA → Deploy → Done
   │         │            │          │             │       │      │
   └──[gate]─┘──[gate]───┘──[gate]──┘──[gate]────┘──[gate]─┘──[gate]─→ Done

Как работает: Процесс формализован как конечный автомат. Переходы — строго детерминированные. Guards — это код, а не «ну, мне кажется, пора». State machine не обманывает, не забывает и не просит прибавку. В отличие от менеджера.

Плюсы: 100% воспроизводимость — каждый раз одно и то же. Ноль токенов сгорает на оркестрацию (guards не спрашивают LLM «а давайте подумаем»). Процесс нарушить невозможно — автомат просто не даст. Легко тестировать. Минусы: Жёсткость. Нельзя сказать «ой, да ладно, пропустим QA, мы спешим». Требует upfront-проектирования. Не подходит для задач, где «результат неизвестен».

Когда подходит: SDLC, compliance, продакшн-процессы. LangGraph, Temporal — ваши друзья. Когда нет: Исследования, креативные задачи, хакатоны — там state machine будет только мешать.

Что берём для Forge

После долгих и мучительных размышлений (и пары чашек кофе) выбор очевиден: State Machine для оркестрации — потому что железобетонные guards не подведут, и Orchestrator+Workers для исполнения — потому что кто-то же должен раздавать указания.

Pipeline не даёт параллелизма (сиди и жди, пока баг-фиксер допишет). Swarm — это анархия, а не архитектура. Monolithic — для тех, кто не верит в будущее.

Именно об этом сочетании — всё, что вы прочитаете дальше.

4. Платформы — ищем того, кто соберёт пазл

Протоколы — это прекрасно, но кто их на практике реализует? Ни одна платформа не даёт MCP + A2A + FSM в одной коробке. Это как заказать пиццу с ананасами, оливками и анчоусами — каждый ингредиент прекрасен по отдельности, но вместе их не кладут. Придётся собирать самим.

4.1 OpenClaw — наш родной «коммуникатор»

Что: Open-source runtime для AI-агентов. Telegram-интерфейс, sessions_spawn, systemEvent, cron.

MCP: Нет нативной поддержки. Агенты вызывают инструменты через shell-команды. A2A: Нет. Агенты могут общаться через sessions_send (proprietary transport). FSM/Workflow: Нет. Процесс — в голове менеджера или в bash-скриптах.

Для чего подходит: Коммуникация (Telegram → агент → человек), спавн агентов, heartbeat-мониторинг. Ограничения: Нет контрактования. Нет типизации. Процесс полностью на усмотрение LLM.

4.2 CrewAI — быстро, дёшево, несердито

Python-фреймворк для быстрого прототипирования. Роли, задачи, процессы (sequential/hierarchical). MCP — через LangChain, A2A — нет, FSM — нет.

Хорош для research и writing-задач. Ограничения: Python-only, persistence нет, гарантий нет. Как прототип из картона — работает, но в дождь не выносить.

4.3 AutoGen — «а давайте поболтаем»

Microsoft. Multi-agent conversation. Group chat. Агенты сидят в чате и обсуждают. MCP — нет, A2A — нет, FSM — нет.

Для брейнштормов — идеально. Для продакшна — агенты болтают, а не работают. Как групповая терапия: всем стало легче, но код не написан.

4.4 LangGraph — граф, который всё помнит

Что: LangChain-экосистема. Направленный граф состояний. Conditional edges, cycles.

MCP: Через LangChain tool calling. A2A: Нет. FSM/Workflow: Да — это основная абстракция. Nodes = функции, edges = переходы.

Для чего подходит: Workflow с условиями, циклами, human-in-the-loop. SDLC как state graph. Ограничения: Python-only. Привязка к LangChain. State в памяти (нужен checkpointer для persistence).

4.5 Dify — «нарисуй мне workflow»

Визуальный workflow builder. Drag-and-drop для тех, кто не хочет писать код. MCP — частично, A2A — нет, визуальный граф — да.

Для нетехнических пользователей — подарок судьбы. Для сложных процессов — ограниченная кастомизация и vendor lock-in.

4.6 Semantic Kernel — Microsoft играет по-крупному

Microsoft. Плагины, планировщики, память. MCP — нативная поддержка. A2A — нет. FSM — через Process Framework (preview).

Enterprise .NET/Python с интеграцией в Azure. Heavy enterprise — не для стартапов, у которых сервер — это MacBook.

4.7 OpenAI Agents SDK — «нейросеть, сделай агента»

OpenAI. Handoffs, guardrails, tracing. MCP — да, нативно. A2A — нет. FSM — упрощённая маршрутизация.

Для тех, кто уже в экосистеме OpenAI. Python-only, persistence — нет. Красиво, быстро, но завтра может измениться — это же OpenAI.

4.8 Google ADK — «A2A из коробки»

Google. Agent Development Kit. A2A — первый класс. Agent Cards, Task delegation, artifacts — всё нативно. MCP — через интеграцию.

Будущий стек для мультиагентных систем с контрактами. Ранний stage, документация скудная. Дерзкий новичок, который может всех обогнать.

4.9 Amazon Bedrock Agents — для тех, кто уже в AWS

AWS. Action groups, knowledge bases, guardrails. MCP — нет, A2A — нет, FSM — упрощённый.

Для тех, кто проснулся в AWS и не может оттуда выбраться. Enterprise compliance — отлично. AWS lock-in, ограниченная кастомизация, дорого — не так хорошо.

5. На чём писать — выбираем молотки и отвёртки

Протоколы выбраны, платформы осмотрены со всех сторон. Осталось решить, чем кодить.

5.1 MCP SDK — стандарт де-факто

Зрелость: ⭐⭐⭐⭐⭐ — официальный SDK от Anthropic. Стабильный, документированный.

Что даёт: Server и Client. JSON-RPC transport. Tool/Resource/Prompt registration. Stdio и SSE transport.

Когда применять: Любой MCP-сервер. GitHub MCP и SDLC MCP — на нём. Когда НЕ применять: Если нужно агент-агент общение (это A2A, не MCP).

5.2 A2A SDK — Google открыл, мир ждёт

Зрелость: ⭐⭐⭐ — референс-имплементация от Google. Работает, но документация минимальна.

Что даёт: Agent Card, Task, Message, Push Notification. JSON-RPC over HTTP/SSE.

Когда применять: Контракты между агентами. Task delegation с обязательными полями. Когда НЕ применять: Если агентов <3 (overkill). Если транспорт уже есть (OpenClaw sessions).

5.3 LangGraph — граф, который ты заслужил

Зрелость: ⭐⭐⭐⭐ — часть LangChain-экосистемы. Активно развивается.

Что даёт: State graph, conditional edges, cycles, human-in-the-loop, checkpointing (SQLite/Postgres).

Когда применять: Workflow с условиями и циклами. SDLC как граф. Human approval steps. Когда НЕ применять: Когда нужен durable execution (при рестарте сервера workflow должен продолжиться). LangGraph checkpointing — не production-grade для этого.

5.4 Temporal — если упасть нельзя

Зрелость: ⭐⭐⭐⭐⭐ — production-grade workflow engine. Используется в Netflix, Coinbase, Datadog.

Что даёт: Durable workflows (переживают рестарт сервера). Retries, timeouts, saga patterns. Go, TypeScript, Python SDK.

Когда применять: Продакшн-процессы, где потеря состояния = катастрофа. CI/CD pipelines. SLA enforcement. Когда НЕ применять: Прототипы. Простые линейные процессы. Когда инфраструктура Temporal server — overhead.

5.5 n8n — «нарисую, блин, автоматизацию»

Зрелость: ⭐⭐⭐⭐ — визуальный automation platform. 400+ интеграций.

Что даёт: Drag-and-drop workflow builder. Webhook triggers. HTTP requests. Code nodes.

Когда применять: Быстрое прототипирование интеграций. Нетехнические пользователи. Простые automation. Когда НЕ применять: Сложная бизнес-логика. Продакшн-процессы с гарантиями. Когда нужен код, а не визуальный редактор.

5.6 Inngest — для тех, кто живёт событиями

Зрелость: ⭐⭐⭐ — event-driven functions platform. TypeScript-first.

Что даёт: Event triggers, durable functions, retries, concurrency control.

Когда применять: Event-driven архитектуры. Serverless. Когда нужно реагировать на события. Когда НЕ применять: Когда важен синхронный workflow. Когда нет TypeScript.

5.7 Custom JSON-RPC — когда хочется своего

Зрелость: ⭐⭐⭐⭐⭐ — это не библиотека, это подход. Пишешь свой протокол поверх JSON-RPC.

Что даёт: Полный контроль. Минимальные зависимости. Можно реализовать за вечер.

Когда применять: Когда стандарты (MCP, A2A) — overhead. Когда нужно быстро. Когда команды 3-5 агентов. Когда НЕ применять: Когда нужна совместимость с другими платформами. Когда есть готовый SDK.

6.3 SDLC = A2A Task Flow

Главная идея: каждый переход фазы SDLC — это не «перекинули комментарий в issue», а полноценный A2A Task с контрактом. Покажу на примере:

{
  "task_id": "sdlc-284-planning-to-design",
  "parent_task_id": "forge-284",
  "from_phase": "Planning",
  "to_phase": "Design",
  "agent_from": "marina",
  "agent_to": "boris",
  "artifacts": {
    "requirements_doc": "docs/specs/284-requirements.md",
    "acceptance_criteria": ["AC1: ...", "AC2: ..."],
    "assignee": "boris",
    "priority": "high"
  },
  "contract": {
    "required_fields": ["requirements_doc", "acceptance_criteria", "assignee", "priority"],
    "validated": true
  }
}

Суть: если contract.validated = false — Task не улетает. Агент-отправитель получает ошибку и идёт фиксить артефакты. Никаких «а, ну я думал, что готово». Gatekeeper переезжает из bash в протокол, где ему и место.

А вот что скрывается за этим контрактом. Конкретный сервер, который знает правила.

6.4 SDLC MCP Server — bash на пенсию

Кастомный MCP-сервер, который упаковывает все правила процесса в MCP Tools. Представьте: вместо того чтобы писать grep "TODO" | if [ $? -eq 0 ], вы просто вызываете sdlc_check_artifact:

Tool Описание Заменяет
sdlc_validate_transition Проверяет можно ли перейти из фазы A в фазу B gatekeeper в CLI-скрипты управления бордом
sdlc_check_artifact Проверяет артефакт на соответствие AC скрипт верификации
sdlc_get_rules Возвращает правила для фазы grep по документация процесса
sdlc_report Генерирует отчёт по пайплайну скрипты мониторинга --daily
sdlc_check_sla Проверяет SLA по приоритету скрипт верификации --no-timeline

Что даёт переход от bash к MCP: - Типизированные параметры — JSON schema вместо «ну там строка, может быть» - Автодокументирование — tool descriptions читаются агентами, а не пылятся в README - Тестируемость — MCP protocol — это стандартный интерфейс - Расширяемость — новое правило = новый tool, а не строчка в grep-пайпе

Хорошо, свой сервер мы написали. Но есть и готовые — например, официальный от GitHub. Бери и пользуйся.

6.5 GitHub MCP Server — ничего не надо писать

Официальный сервер от GitHub. Тoolset «Projects» (доступен с октября 2025). Просто подключаешь — и CRUD-операции с бордом работают. Вот что он закрывает:

Tool Отправляет на пенсию
list_projects CLI-скрипты управления бордом
get_project_fields Их же (поля)
get_project_items Их же (items)
add_project_item Их же (добавление)
update_project_item Их же (обновление)

update_issue gh issue edit

Подарок: MCP server сам управляет кэшированием. локальный кэш можно удалять с чистой совестью.

9. А как же OpenClaw?

Критический вопрос: OpenClaw уже умеет спавнить агентов (sessions_spawn), слать события (systemEvent) и общаться (sessions_send). Нам точно нужен A2A поверх этого?

Хороший вопрос. Рассмотрим варианты:

Вариант A: A2A нативно — агенты общаются по A2A, OpenClaw — только транспорт. Требует отдельного A2A-сервера. Красиво, но тяжеловато для первого шага.

Вариант B: A2A-семантика на OpenClaw — контракты и Agent Cards как в A2A, но транспорт через OpenClaw sessions. Меньше инфраструктуры при той же формализации.

Вариант C: MCP-only сначала — GitHub MCP Server + SDLC MCP Server. Агенты через MCP работают с бордом, gatekeeper через MCP tools. A2A — потом.

Вывод: Вариант C. MCP решает 80% боли (CRUD + gates). A2A — правильное, но не срочное дополнение.

10. План действий — меньше слов, больше дела

Хватит теории. Вот конкретика с дедлайнами. Первый результат — через 2-3 дня. Если за неделю ничего не изменилось — возвращайтесь к bash и не мучайтесь.

🔧 Фаза 1: MCP для CRUD (2-3 дня)

  • Развернуть GitHub MCP Server (официальный, Docker)
  • Протестировать Projects V2 toolset на реальном борде
  • Проверить coverage: все ли поля читаются и пишутся
  • Перевести CRUD-операции на MCP
  • Обновить мониторинг

🏗️ Фаза 2: SDLC MCP Server (3-5 дней)

  • Написать кастомный MCP Server (Python SDK)
  • Правила процесса как MCP Tools
  • Gatekeeper → sdlc_validate_transition
  • Верификация → sdlc_check_artifact
  • Написать тесты (MCP protocol test suite)

🤝 Фаза 3: A2A для контрактов (5-7 дней)

  • Определить Agent Cards
  • Реализовать A2A Task delegation
  • Контракты переходов фаз как обязательные поля Task
  • Интеграция с OpenClaw sessions

🚀 Фаза 4: AGP — на вырост (по потребности)

  • Актуально при 50+ агентах и межорганизационном взаимодействии
  • Пока — не заморачиваемся

11. А если нет? — альтернативы для скептиков

Всё выше — идеальный сценарий. А если MCP + A2A — это перебор для ваших задач? Вот что можно вместо этого:

  1. github-project-mcp (PyPI) — готовый MCP-сервер для GitHub Projects V2. Без Docker, одна команда pip install
  2. gh CLI MCP wrapper — обёртка над gh CLI как MCP-сервер (codingbutterbot/gh_cli_mcp). Если не хотите уходить от консоли
  3. OpenClaw native tools — остаться как есть, но формализовать контракты в JSON-схемах. Без новых протоколов, без новых зависимостей. Консервативно, но надёжно

12. Вердикт

Архитектурное решение — за вами. Выбор между башем XXI века и протоколами, придуманными умными людьми за вас. Если у вас 3 агента и один репозиторий — оставайтесь на bash, проживёте. Если 15 — протоколы окупятся за первую же неделю, когда агент №12 случайно не затёрт код агента №3.

Как говорил один мудрый архитектор: «Выбирайте не то, что модно. Выбирайте то, от чего не будет больно на ретроспективе».
